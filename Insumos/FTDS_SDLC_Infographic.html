FTDS Engineering · Software Development Life Cycle

How a change travels through FTDS —
from a developer's laptop to Production

The path every code change follows: build locally, prove it through two mandatory automatic gates, run optional deeper checks, merge, then survive nightly / system testing and manual QA before release. Each stage names the AKS cluster that does the work.

Where it stalls · escalation

The five load-bearing clusters are 01, 02, 04, 07, 09. If 02 or 07 go red the whole program stops merging — that's the call to Tim Florenki, with Rahul on the program impact. If the nightly (01 / 09) breaks, that's Andrea on the test side.

FinOps lens · biggest lever

Cluster 04 alone runs ~$24K/month (~$285K/yr) and climbing, driven by overlapping on-demand subsets re-running the same tests. The optimization isn't less testing — it's de-duplicating subsets and parking idle CI clusters (≈15 min to spin back up).

Read it this way: a change only reaches Production by clearing every phase left to right. Phase 2's two gates are the only mandatory automatic stops — everything in Phase 1 and the on-demand checks are the developer's judgment. The clusters are the same physical fleet the FinOps program monitors, which is why criticality and cost sit on the same map.
